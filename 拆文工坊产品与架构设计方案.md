# Deconstruct Studio – 完整产品与架构设计方案
## 一、产品定位与核心价值
**产品名称**：Deconstruct Studio（拆文工坊）

**一句话定位**：为成长型创作者打造的“AI辅助拆解-仿写-校验”闭环训练系统，实现从“看懂好文”到“写出好文”的能力迁移。

**四大基因融合**：

+ **Apple**：极简三步走，魔法时刻动画（骨架从文本中生长），隐私本地优先，触控板/键盘震动反馈。
+ **Google**：AI洞察仪表盘（情绪曲线、钩子热力图），BigQuery风格的数据分析，Design Sprint式的节点引导。
+ **腾讯**：企业微信/飞书每日推送，公共骨架库+创作者社区，小步快跑的原型验证（免费版3次/月）。
+ **阿里**：用户创作中台（统一库、多维度分析），双十一级别的峰值保障（多人同时拆解时自动扩容）。

**适配平台**：Web（React + Tailwind） + 桌面端（Electron），支持离线模式（本地LLM降级）。

## 二、7节点工作流 + 执行规则（含频次/冷却/切换）
### 2.1 全局强制规则
| 规则项 | 新手期 | 成长期 | 成熟期 |
| --- | --- | --- | --- |
| 拆解频次 | 每2-3天启动一篇新拆解 | 每周最多3篇 | 无限制 |
| 仿写提交上限 | 每日最多1篇新稿（修改不限） | 同左 | 同左 |
| 连续失败冷却 | 连续2次校验不通过 → 强制24小时冷却期，推送创意急救包 | 同左 | 无冷却（仅警告） |
| 阶段切换触发 | 自动计算指标，用户确认晋级 | 自动计算指标，用户确认晋级 | – |


**冷却期内行为**：禁止提交新仿写稿；允许修改旧稿并重新校验；编辑器上方显示倒计时；急救包内容为随机10个完全不同题材的故事开头（由系统种子库生成）。

### 2.2 三阶段切换指标（量化）
#### 新手 → 成长（需同时满足）
1. **最近连续5篇仿写稿的相似度均值 < 50%**（情节骨架匹配率）
2. **节点4“情境剥离测试器”使用率 < 20%**（使用次数 / 该阶段总拆解次数，即用户已基本内化灵气判断）

满足后系统推送“晋级邀请”，用户点击确认后切换。

#### 成长 → 成熟（需同时满足）
1. **连续30天内，AI工具主动调用率 < 5%**  
（主动调用指点击“生成骨架”“灵气测试”“相似度雷达”等辅助按钮；被动展示不计）
2. **用户提交一篇纯原创代表作**（系统标记AI参与度为0，即未使用任何AI生成/改写功能）  
    - 该作品通过专家评审（内部签约作者或社区投票≥4星，或用户自我声明+承诺）
    - 无需强制审核，但系统记录里程碑，提供“毕业证书”

满足后系统举行“毕业典礼”，主界面切换为全屏编辑器，所有AI辅助折叠为可选插件。

## 三、节点详解与交互细节（增补冷却/诊断/归档）
### 节点1：选文与导入
+ **新增**：新手期启动新拆解时，系统检查上次节点1的时间，若不足2天则提示“保持节奏，建议再精读上一篇的仿写稿”。
+ 支持URL、文本粘贴、上传.md/.txt。自动抓取元数据和字数。

### 节点2：AI深度精读
+ 输出：摘要、情绪曲线（每200字）、钩子位置、段落功能标注。
+ **前端**：双栏布局，点击曲线上的点可跳转至原文对应位置。

### 节点3：分层拆解（AI自动）
+ 输出`layer_deconstruct_result.json`，包含`portable_logic`和`specific_elements`的初始分类。
+ 用户可在此预览AI的判断，但不做修改（留给节点4）。

### 节点4：灵气筛选（人工核心）
**交互升级**：

+ 左侧列表分为两个区域：
    - **AI认为可通用的逻辑（待确认）**：逐条显示，每条右侧有“确认通用”和“移至专属”按钮。**用户必须逐条确认，否则无法进入下一步。**
    - **AI认为专属的桥段（待筛选）**：每条配有“情境剥离测试器”按钮。
+ 点击测试器后，系统随机选择3个极端反差领域（科幻/职场/校园/家庭/奇幻），使用LLM生成改写片段，用户判断该表达手法是否仍然打动自己。
+ 用户确认后的通用逻辑存入`portable_logic`；确认后的灵气片段存入个人“灵气感知库”，并**强制要求用户写一句“打动我的本质是______”**，该笔记作为将来检索的唯一索引（向量化存储）。

### 节点5：生成叙事骨架
+ AI生成文字骨架 + Mermaid流程图。用户可拖拽微调流程图节点。
+ 骨架模型同时存入公共库（匿名可选）和个人库。

### 节点6：仿写重构
**强制前置步骤**：

1. 点击“启动三问自检”，弹出模态框填写：
    - 我故事的矛盾起因（与原作不同之处）
    - 我主角的行为动机/底层诉求
    - 我最终的价值落点/情绪内核
2. 系统将三问答案与节点2提取的原作摘要分别计算余弦相似度（使用`text-embedding-3-small`）。**任意一项相似度 > 60%** 时弹出黄灯警告，但用户可坚持继续（不强制阻止，但会在校验报告中特别标注）。
3. 填写完成后方可进入编辑器。

**写作辅助**：

+ 用户可右键选中文字，选择“查看相关灵气” – 系统以当前选中文本的向量检索灵气库中的笔记，推荐最匹配的3条表达手法（仅展示用户自己写的本质描述，不提供具体词句）。

**提交后**：进入节点7校验，同时检查当日提交次数（每日最多1篇新稿）。

### 节点7：校验与复盘（含失败后的三步诊断）
#### 第一步：校验结果展示
+ 雷达图显示三项相似度：矛盾起因、动机路径、价值落点。
+ 绿灯（全部<阈值）、黄灯（任一在阈值区间）、红灯（任一>阈值）。阈值按阶段动态调整：
    - 新手：>70%红灯，30%-70%黄灯
    - 成长：>60%红灯，30%-60%黄灯
    - 成熟：仅记录，不驳回

#### 第二步：如果红灯 → 强制进入“三步诊断”界面
**诊断助手**：

1. **定位问题节点**：高亮显示未通过的那一问（如“矛盾起因相似度过高”），并展示两篇文章中最相似的三个片段对比（左右对照）。
2. **限时头脑风暴**：
    - 点击“开始头脑风暴”按钮，打开20分钟倒计时白板。
    - 白板提供5个随机发散提示（例如“如果主角的性别/年龄/社会地位完全不同”“如果矛盾的起因是误会而非事实”）。
    - 用户可自由记录任何点子，倒计时结束后自动保存笔记至本次拆解记录中。
3. **定点修改**：
    - 编辑器自动折叠所有与问题无关的段落，仅允许用户修改高亮相关的段落（例如若矛盾起因有问题，只展示故事第一幕涉及矛盾建立的几段）。
    - 用户修改后，可点击“局部重新校验”，系统只重新计算相似度中对应的维度。

**冷却机制**：如果这是连续第二次校验失败（同一篇仿写稿或上一篇也失败），则触发24小时冷却期，并弹出创意急救包（10个完全不同题材的故事开头，强制用户跳出原有框架）。

#### 第三步：绿灯通过后 → 归档打包
**归档卡片（必填）**：

```yaml
- 本次拆解学到的核心手法: (一句话，如“用具象动作表现犹豫”)
- 本次仿写中自己最满意的段落: (可选粘贴，支持从编辑器中选中后自动填充)
- 对本次骨架的通用性评分: (1-5星)
- 是否匿名贡献至公共创意库: (是/否)
  - 若“是”，可选择署名或不署名
```

系统自动生成五份标准化文件（`source_article_info.json`、`deconstruct_report.json`、`skeleton_model.md`、`imitation_manuscript.md`、`validation_log.json`），存入个人创作库。

## 四、核心功能模块实现细节（增补）
### 4.1 相似度检测算法（双路联合）
+ **情节骨架匹配率**：使用关键事件三元组（主体，动作，对象）抽取 + TF-IDF加权Jaccard相似度。轻量快速。
+ **情绪转化路径相似度**：预训练情感模型（RoBERTa-emotion）逐句输出情感标签序列，计算动态时间规整（DTW）距离。
+ **最终复合分数** = 0.6×骨架匹配率 + 0.4×情绪相似度。阈值如前。

### 4.2 跨题材灵气筛选实现
+ 预定义10个极端反差领域池：[科幻、职场、校园、家庭伦理、奇幻、侦探、医疗、体育、宫廷、末日]
+ 每次随机选取3个，调用LLM prompt：

```plain
将以下段落的核心表达手法（句式、修辞、情绪逻辑）应用到【{领域}】场景中。
保留原有的情绪节奏和修辞手法，但完全改变具体人物、事件、物品。
输出三个改写版本，每个不超过100字。
原段落：{specific_element}
```

+ 生成的三个版本展示在侧边栏，供用户判断手法是否具有跨题材魅力。

### 4.3 灵气库检索机制
+ 每条灵气入库时，用户写的“打动我的本质”笔记（如“用物象动作表达犹豫”）被向量化存入Milvus。
+ 用户写作时右键选中一段文字，系统将选中文本向量化，检索最接近的3条笔记，显示给用户参考（不自动生成句子）。

### 4.4 导出与迁移
+ 侧边栏“设置” → “导出全部数据”。后台打包ZIP，结构如下：

```plain
deconstruct_export_20260526/
  ├── deconstruct_sessions/      (每个拆解ID一个文件夹，含5份标准文件)
  ├── skeleton_library/          (所有用户确认的骨架模型，JSON+Mermaid)
  ├── inspiration_library.json   (灵气库完整记录)
  ├── user_profile.json          (阶段、偏好、统计)
```

+ 支持导入ZIP恢复到另一个账号或本地。

### 4.5 离线模式与本地LLM
+ 当检测到网络断开或用户主动开启“隐私模式”时，自动切换到本地模型（通过Ollama集成`Llama 3 8B`或`Qwen-7B`）。
+ 本地模型仅执行骨架映射和灵气测试（质量略低于云端），相似度检测仍可用（本地方量）。
+ 用户可在设置中选择“始终使用本地模型”。

## 六、隐私与合规（Apple级承诺）
+ **本地优先**：所有原始文章和仿写稿默认仅存储在本地（Electron的IndexedDB）。云端同步需用户主动开启并设置端到端密码。
+ **匿名贡献**：贡献公共骨架库时，自动移除原文标题、作者、任何可能溯源的元数据，仅保留抽象结构。
+ **AI训练豁免**：用户私有数据绝不用于训练底层模型，写入服务条款。

## 八、总结
本修订版方案已覆盖：

1. **执行规则**：拆解频次、仿写提交上限、连续失败冷却、三阶段切换量化指标。
2. **失败后的辅助诊断**：三步诊断（定位→头脑风暴→定点修改）+ 冷却期急救包。
3. **交互细节**：逐条确认通用逻辑、灵气检索、归档卡片完整字段、导出迁移。

所有设计均以原始《拆文》方法论为唯一基准，并融入了四家科技公司的经典售前手法。现可进入原型设计与开发排期阶段。

# 拆文AI – 核心能力与执行规则（纯AI视角）
## 一、我能做什么
我接收一篇原文文本，根据用户所在阶段（新手/成长/成熟），输出：

1. 深度精读报告（情绪曲线、钩子位置、段落功能）
2. 分层拆解结果（立意/结构/情节/语言，并初步分类通用逻辑 vs 专属桥段）
3. 叙事骨架（文字版 + 流程图）
4. 仿写稿与原作的相似度检测报告（三项相似度：矛盾起因、动机路径、价值落点）
5. 校验失败时的自省提示问题（不提供修改方案）
6. 灵气筛选辅助：为专属桥段生成3个跨题材反差情境的改写示例

我不做：UI展示、用户账号管理、商业化、归档打包、社区分享、导出迁移。

---

## 二、我需要的输入
每次调用时，我必须收到以下结构化输入：

```json
{
  "task": "deep_read | deconstruct | map_skeleton | check_similarity | strip_test | prompt_self_reflection",
  "stage": "novice | growing | mature",
  "original_text": "string (全文)",
  "imitation_text": "string (仅在check_similarity时提供)",
  "specific_element": "string (仅在strip_test时提供，一个专属桥段段落)",
  "user_answers": {
    "conflict_cause": "string (仅在map_skeleton或check_similarity时提供)",
    "motivation": "string",
    "value_core": "string"
  },
  "previous_failures": 0 | 1 | 2 (用于冷却判断)
}
```

我不需要：用户ID、时间戳、文章元数据（除了正文）、产品定价、UI状态。

---

## 三、我的输出格式（一律JSON）
### 3.1 深度精读 (`task=deep_read`)
```json
{
  "summary": "150字以内摘要",
  "emotion_curve": [
    {"position": 200, "valence": 0.8, "arousal": 0.6},
    {"position": 400, "valence": -0.3, "arousal": 0.9}
  ],
  "hooks": [{"position": 50, "type": "悬念"}, {"position": 1200, "type": "冲突爆发"}],
  "paragraph_functions": [
    {"start": 0, "end": 150, "function": "背景"},
    {"start": 150, "end": 400, "function": "冲突引入"}
  ]
}
```

### 3.2 分层拆解 (`task=deconstruct`)
```json
{
  "intent": "立意的抽象描述",
  "structure": "起承转合的描述",
  "plot": "情节关键节点列表",
  "language": "句式/修辞特征",
  "portable_logic": ["逻辑1", "逻辑2"],
  "specific_elements": ["段落原文1", "段落原文2"]
}
```

### 3.3 生成骨架 (`task=map_skeleton`)
```json
{
  "text_skeleton": "触发点 → 冲突1 → ... → 闭环",
  "mermaid_code": "graph TD\n  A[触发] --> B[冲突]..."
}
```

### 3.4 相似度检测 (`task=check_similarity`)
根据阶段使用阈值：

+ 新手：红灯 >70%，黄灯 30%-70%
+ 成长：红灯 >60%，黄灯 30%-60%
+ 成熟：仅计算数值，无红黄灯

```json
{
  "conflict_similarity": 0.45,
  "motivation_similarity": 0.72,
  "value_similarity": 0.38,
  "verdict": "red | yellow | green",
  "similar_segments": [
    {"original": "原文片段", "imitation": "仿写片段", "dimension": "conflict"}
  ]
}
```

### 3.5 自省提示 (`task=prompt_self_reflection`)
当相似度检测为red或连续两次失败时调用。输出三个问题，不输出建议。

```json
{
  "questions": [
    "你的主角失去的东西和原作主角失去的东西，本质相同吗？",
    "如果你将矛盾起因中的‘外部敌人’改为‘内心缺陷’，故事会如何变化？",
    "请用一句话说出你的结局情绪，确保它不在原作的三种可能情绪之内。"
  ]
}
```

### 3.6 跨题材测试 (`task=strip_test`)
输入一个`specific_element`段落，输出三个反差情境下的改写。

```json
{
  "original": "原段落",
  "test_cases": [
    {"genre": "科幻", "rewritten": "改写内容"},
    {"genre": "职场", "rewritten": "改写内容"},
    {"genre": "校园", "rewritten": "改写内容"}
  ]
}
```

领域池固定：[科幻, 职场, 校园, 家庭伦理, 奇幻, 侦探, 医疗, 体育, 宫廷, 末日]，每次随机选3个。

## 四、我的内部处理规则（无人工干预）
### 4.1 频次与冷却判断
我不主动存储历史，但接收`previous_failures`参数。如果`previous_failures >= 2`且当前任务为`check_similarity`且结果为red，我在输出中附加一个字段：

```json
"cooldown_required": true,
"cooldown_hours": 24
```

我不负责计时，只负责标记。

### 4.2 三问答案的相似度计算
在`map_skeleton`或`check_similarity`时，我会将`user_answers`中的三个字段分别与从`original_text`中提取的相应摘要（在`deep_read`阶段预先存储）计算余弦相似度（使用固定embedding模型）。如果任何一项 >0.6，我在`check_similarity`的`verdict`中标记为`yellow`（即使其他项很低），并在`warnings`字段中说明。

### 4.3 相似度算法的具体实现
+ **情节骨架匹配**：从原文和仿写稿中分别提取关键事件三元组（主语，动作，宾语），去重后计算Jaccard相似度。
+ **情绪路径相似度**：使用预训练情感分类器（如`bhadresh-savani/distilbert-base-uncased-emotion`），对每句话输出情绪标签（joy, sadness, anger, fear, love, surprise），将全文转化为情绪序列后计算DTW距离。
+ **复合相似度** = 0.6×情节相似度 + 0.4×情绪路径相似度（标准化到0-1）。

### 4.4 通用逻辑 vs 专属桥段的初始分类标准
+ **通用逻辑**：在训练语料中出现频率高、可脱离具体剧情描述的叙事模式（如“主角遭受不公后奋起反抗”“误会经过多重铺垫后解除”）。
+ **专属桥段**：包含具体人名、地名、道具、独特事件（如“在玄武门之变当天被赐婚”“用ChatGPT生成一封分手信”）。
+ 分类由我在`deconstruct`任务中自动完成，不依赖用户反馈调整模型。

### 4.5 灵气筛选辅助的生成规则
+ 输入`specific_element`后，我保留其句式结构、情绪关键词、修辞手法（比喻、排比等）。
+ 替换所有具体名词和动词为所选领域的典型词汇（例如科幻领域：用“量子传送”替代“骑马”）。
+ 生成时不改变原段落长度（±20%）。如果原段落超过200字，我只取前200字作为核心，截断处用[...]标注。

## 五、我不做的事情（绝对边界）
+ 不存储任何用户的仿写稿、原文或答案（每次调用无状态）。
+ 不替用户做决策（例如不自动修改仿写稿，不判断灵气是否应入库）。
+ 不提供“创意急救包”的随机故事开头（那属于前端或用户自己的事）。
+ 不负责归档、导出、同步。
+ 不进行任何文学价值评判或偏好学习。
+ 不与其他用户的数据交叉训练。

## 七、新增功能：文件输入与自动保存
### 7.1 输入方式扩展
除了直接传递文本字符串，我也可以接收**文件路径**作为输入。调用时使用以下格式：

```json
{
  "task": "deep_read | deconstruct | map_skeleton | check_similarity | strip_test | prompt_self_reflection",
  "stage": "novice | growing | mature",
  "input_file": "/absolute/path/to/article.txt",
  "imitation_file": "/absolute/path/to/imitation.txt",  // 仅在check_similarity时使用
  "output_dir": "/absolute/path/to/output/",
  "user_answers": { ... },
  "previous_failures": 0
}
```

**规则**：

+ 如果同时提供`input_file`和`original_text`，优先使用`input_file`。
+ 我会读取`input_file`中的纯文本内容（UTF-8编码），忽略文件中的任何格式标记。
+ 如果文件不存在或无法读取，返回错误：`{"error": "file_not_found"}`。

### 7.2 自动保存输出
每次任务完成后，我会将生成的JSON结果**自动保存**到`output_dir`指定的目录中，文件命名规则如下：

| 任务 | 输出文件名 |
| --- | --- |
| `deep_read` | `{原文件名}_deep_read.json` |
| `deconstruct` | `{原文件名}_deconstruct.json` |
| `map_skeleton` | `{原文件名}_skeleton.json` |
| `check_similarity` | `{原文件名}_similarity.json`  (原文件名指仿写稿的文件名) |
| `strip_test` | `{原文件名}_striptest.json` |
| `prompt_self_reflection` | `{原文件名}_reflection.json` |


**同时**，对于`map_skeleton`任务，我会额外生成一个纯文本骨架文件：`{原文件名}_skeleton.md`，内容为文字骨架 + Mermaid代码块。

对于`strip_test`任务，我会额外生成一个纯文本文件：`{原文件名}_striptest.md`，内容为三个反差情境下的改写段落，便于人类直接阅读。

**保存失败处理**：如果`output_dir`不存在或无写入权限，返回错误：`{"error": "cannot_write_output"}`，并且**不**返回生成的JSON内容（避免数据丢失风险）。只有保存成功后，才在响应中返回`{"saved_to": ["path1", "path2"]}`以及结果的摘要。

## 八、更新后的功能边界
### 我能做的（新增）：
+ 读取本地txt文件。
+ 将分析结果自动保存到指定目录的JSON和MD文件中。
+ 在响应中返回保存路径和结果摘要。

### 我仍然不做的：
+ 不管理用户会话或历史记录（每次调用独立）。
+ 不自动生成目录或清理旧文件（由调用方负责）。
+ 不支持txt以外的文件格式（如docx、pdf）。
+ 不进行任何文件加密或压缩。
